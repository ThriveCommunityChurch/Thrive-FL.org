# Thrive-FL.org — Developer Guide

**Purpose:** Public marketing website and sermon browsing platform for Thrive Community Church (Estero, FL). Static/SSG-first design for performance and SEO.

## Tech Stack

- **Framework:** Next.js 16+ (App Router)
- **Language:** TypeScript 5.6+
- **Styling:** CSS Modules + CSS (app.css, critical.css for above-the-fold)
- **Icons:** FontAwesome 7.1+, Simple Icons
- **Audio:** Howler.js 2.2+ (audio player for sermon downloads)
- **Content Delivery:** Mailgun (contact form emails)
- **API Integration:** Thrive Church Official API for sermon data, events, etc
- **Deployment:** AWS Amplify (Next.js SSG export)

## Quick Start

### Local Development

```bash
# Install dependencies
npm install

# Start dev server (runs on localhost:3000)
npm run dev

# Navigate to http://localhost:3000
```

Development server includes hot reload. Page changes reflect immediately.

### Build & Deployment

```bash
# Build for production (static export)
npm run build

# Start production server (for testing)
npm start

# Linting
npm run lint
```

**Production deployment:** Branches to `dev` trigger automatic AWS Amplify builds and deploy to staging. Merges to `master` deploy to production.

## Key Features

### Sermon Browsing (`/sermons`)
- Fetches sermon series from ThriveChurchOfficialAPI
- Displays sermon metadata (date, speaker, summary, tags)
- Links to sermon audio via Howler.js player
- SEO-optimized with meta tags for sermon content

### Livestream (`/live`)
- Real-time stream state from API
- Stream status indicator (live/offline)
- Embedded video player or link to external platform

### Contact Form (`/contact`)
- Validates user input (TypeScript)
- Submits via Mailgun API
- Error handling with user feedback

### Blog (`/blog`)
- Church news and announcements
- Markdown or API-sourced content
- Searchable and filterable

## Data Integration

### API Calls
All API calls to ThriveChurchOfficialAPI happen in `app/lib/` utilities. Example pattern:

```typescript
// app/lib/api.ts
export async function getSermons() {
  const res = await fetch(`${API_BASE_URL}/sermons`, {
    headers: { Accept: 'application/json' },
  });
  if (!res.ok) throw new Error('Failed to fetch sermons');
  return res.json();
}
```

Page components import and call these utilities. Data fetching happens at **build time** (static generation) or **request time** (dynamic routes).

### Content Management
- **Sermon Data:** ThriveChurchOfficialAPI (read-only)
- **Church Info & Contact:** Static in `app/layout.tsx` or `public/config.json`
- **Blog/News:** Can be static Markdown files in `app/blog/` or API-sourced

## Static Generation & SEO

### Static Exports
Next.js generates static HTML at build time for maximum performance:

```bash
npm run build  # Exports to .next/static/
```

All pages are **pre-rendered as HTML** and served from a CDN.

### SEO Optimization
- **Meta Tags:** Use Next.js Metadata API in each page's layout/component
- **Structured Data:** JSON-LD for sermons, events, organization
- **Sitemap:** Auto-generated by `app/sitemap.ts`
- **Robots.txt:** Auto-generated by `app/robots.ts`
- **OpenGraph:** Social media preview tags for sermons/events

Example (metadata in page or layout):

```typescript
export const metadata = {
  title: 'Thrive Community Church Sermons',
  description: 'Watch and listen to sermons from Thrive Church.',
  openGraph: {
    title: 'Thrive Sermons',
    description: 'Inspiring messages for spiritual growth.',
  },
};
```

### Mobile Responsive
All components use responsive CSS (mobile-first). Test on multiple screen sizes before deployment.

## Code Standards

### TypeScript
Strict mode enabled. All files are `.ts` or `.tsx`. Reference:
- **Naming conventions:** See `Docs/Shared/Naming-Conventions.md`
- **Type safety:** Use explicit return types and avoid `any`

### Styling
- **CSS Modules:** Component-scoped styles in `app/components/ComponentName.module.css`
- **Global Styles:** `app/app.css` for global resets and utilities
- **Above-the-Fold:** `app/critical.css` for inline critical path styles (improves LCP metric)

### Components
- Functional components with hooks only (no class components)
- Props typed with `interface` or `type`
- One component per file
- Example: `app/components/SermonCard.tsx`

### Performance
- Lazy load images with `<Image>` from Next.js
- Code split large components with `React.lazy()` + `Suspense`
- Monitor Core Web Vitals (LCP, CLS, FID)

## Worktree Discipline

Always use project-local worktrees in `.worktrees/` for feature branches:

```bash
git worktree add .worktrees/feature-sermon-filters origin/dev
```

**Never use global worktrees.** The orchestrator (main Claude session) creates the worktree before spawning implementer agents. Pass the full absolute worktree path in the agent prompt.

## Testing

Minimal testing for marketing site (static content priority). If adding interactivity, test in dev:

```bash
npm run dev
# Manual testing: navigate pages, check links, test contact form
```

No automated test suite currently configured. Focus on:
- Links are valid
- API calls don't error
- Forms submit correctly
- Mobile responsiveness

## Environment Variables

If API secrets needed (e.g., Mailgun API key):

```
# .env.local (git-ignored)
NEXT_PUBLIC_API_BASE_URL=https://api.church.com
MAILGUN_API_KEY=xxx
MAILGUN_DOMAIN=xxx
```

**Note:** Only `NEXT_PUBLIC_*` variables are accessible in browser. Sensitive keys stay server-side.

## Deployment

### AWS Amplify
- **Staging:** Auto-deploy on push to `dev`
- **Production:** Auto-deploy on push to `master`

Amplify runs:
```bash
npm run build
npm run start
```

and exports static files to CloudFront CDN.

### Before Merging to Production
1. Test locally: `npm run dev` and `npm run build`
2. Verify all links work
3. Check mobile responsiveness
4. Validate contact form functionality
5. Test sermon playback and download
6. Verify meta tags and SEO with browser dev tools

## Key Architectural Constraints

- **Static-first:** Prefer SSG; dynamic routes only when necessary
- **No server state:** All data sourced from API or static files
- **API-driven content:** Sermon data, events, blog posts fetched from ThriveChurchOfficialAPI
- **No custom auth:** Marketing site is public; no login required
- **Minimal dependencies:** Keep bundle small for CDN delivery
- **Performance:** Optimize for Core Web Vitals (LCP < 2.5s, CLS < 0.1)
